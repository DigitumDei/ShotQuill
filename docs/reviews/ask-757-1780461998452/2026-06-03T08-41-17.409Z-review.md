### Claude (claude / claude-sonnet-4-6)

```text
**Blocking Issues**

**Float-to-string in normalized mask coordinates is fragile across Kotlin Multiplatform targets.**

`PhotoEditPromptAssembler.kt` does:
```kotlin
val right = bounds.left + bounds.width
val bottom = bounds.top + bounds.height
append("... ${bounds.left} to $right horizontally and ${bounds.top} to $bottom vertically ...")
```

On Kotlin/JS, `Float` arithmetic is backed by 64-bit doubles. The nearest 32-bit float to `0.1f` is `≈ 0.10000000149011612`. When added to `0.5f` in double precision (as Kotlin/JS does), the result is `≈ 0.60000000149011612`. JavaScript's `Number.toString()` will render this as `"0.60000000149011612"`, not `"0.6"`. The test `includesMaskRegionWhenPresent` asserts `"0.1 to 0.6 vertically"` which will not match. This is a real failure on Kotlin/JS; `commonTest` runs there. The fix is to format with explicit decimal precision (e.g. `"%.6f".format(value)` or truncation to a fixed number of significant figures) rather than raw string interpolation of float arithmetic results. The issue exists in both the assembler's output to the AI and in the test.

---

**Non-Blocking Issues**

**`aspectRatio != null` is used as proxy for dimension availability.** The old `buildEditPrompt` guarded `recommendedWidthPx` and `recommendedHeightPx` independently:
```kotlin
if (preset.recommendedWidthPx != null && preset.recommendedHeightPx != null) { ... }
```
The assembler now writes `${preset.recommendedWidthPx}x${preset.recommendedHeightPx}px` inside the `aspectRatio != null` branch, with no null guard. The current preset data happens to be consistent — every preset with a non-null aspect ratio also has non-null dimensions — but `PlatformPreset` permits the combination `(aspectRatio=X, recommendedWidthPx=null)`. A future platform addition would silently produce `"nullxnullpx"` in every AI prompt. Worth restoring the independent null check.

**`Custom` intent wording is redundant.** `EditIntent.Custom.promptIntent` = `"Follow the user's custom editing instructions."` The assembled output is:
> "Edit this image: Follow the user's custom editing instructions. [actual instructions]."

The first sentence adds nothing when the instructions immediately follow. This is a model-facing string, so the noise may degrade prompt quality slightly.

**Missing newline at end of file** in both `PhotoEditPromptAssembler.kt` and `PhotoEditPromptAssemblerTest.kt`. Minor, but will cause diffs to show up incorrectly in some editors and may fail linting.

**`stablePromptSummary` truncates at 72 characters; the test relies on this silently.** The test for `fakeProviderEditPhotoSummaryDerivedFromAssembledPrompt` asserts `summary.contains("Edit this image:")`. This works today because `stablePromptSummary` takes the first 72 characters, and `"Edit this image:"` (16 chars) appears before the cutoff. ---

**Strong Concerns**

**Float precision in AI-facing text, not just in tests.** The assembler embeds raw `Float.toString()` of arithmetic results in the prompt string sent to OpenAI. A coordinate like `"0.60000000149011612"` is technically interpretable, but it signals to the model that the caller may not have full control of its output, which can degrade generation quality. This needs explicit formatting regardless of the test platform issue.

Now it is secondary text appended after the intent description. The field name "prompt" now misleads: it is not a prompt, it is user notes or a short directive that augments the assembled prompt. No rename, no deprecation, no doc update. Any caller that previously treated `prompt` as the complete AI instruction is now in a different contract silently. The ViewModel test's circular equality check (`assemble(storedRequest) == storedPrompt`) does not catch a caller that passes the wrong text in this field.
