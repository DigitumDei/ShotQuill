## Executive Summary

This branch introduces a FinalPostComposerViewModel and screen that let users review, edit, and share a post, backed by a new `final_post_content` table (schema v3) and Android-specific share/clipboard launchers. The settled product decision for this branch is that `Exported` and `Shared` mean the post has been successfully handed off to the Android OS chooser; that chooser handoff is the completion signal for durable workflow state in this implementation. The remaining workflow-integrity defect is the blank `editedCaption` path, which can re-enable Share after any state rebuild via `toState()` because `toState` checks `!= null` while `updateCaption` checks `isNotBlank()`. There is also an Android-launcher robustness issue around FileProvider crashes and image-stream degradation. Those issues keep the verdict at revise.

## Product Decision

For this branch, `Exported` and `Shared` mean the post has been successfully handed off to the Android OS chooser. That handoff is the completion signal for durable workflow state in this implementation: once the chooser opens successfully, the export is treated as exported and the draft is treated as shared.

This does not mean the user completed an external share in the target app, because this flow cannot observe chooser cancellation or downstream target-app completion. Future review comments and follow-up work should treat chooser handoff as the canonical definition unless a separate product decision explicitly changes it.

## Blocking Issues

- Blank edited caption bypasses share guard after state rebuild (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/FinalPostComposerViewModel.kt): updateCaption() gates canShare on text.isNotBlank(), so clearing the caption disables Share in the current session. However, toState() and all photo-selection paths compute canShare via effectiveCaption(content) != null. A persisted editedCaption="" satisfies != null, so any reload, selectOriginalPhoto(), or selectEditedPhoto() call re-enables Share with an empty caption. shareOrExport() trusts state.actions.canShare without re-validating. Both reviewers confirmed the mechanism; Codex rated it blocking and Claude validated it as a complete bypass of the share guard.
- AndroidPostShareLauncher FileProvider crash outside try/catch and silent image-stream degradation (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/share/AndroidPostShareLauncher.kt): FileProvider.getUriForFile() is called inside the intent builder's apply block before the try/catch at line 30, so an unmapped path throws IllegalArgumentException and crashes the app before share() can return false. Separately, when imageUri is non-null but the file no longer exists, the intent is sent with type image/* but no EXTRA_STREAM; startActivity can succeed, and the ViewModel still records an Exported image post with no image actually shared. Both reviewers confirmed both sub-claims.

## Non-Blocking Issues

- Per-keystroke database writes on the UI thread, outside the operation mutex (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/FinalPostComposerScreen.kt): updateCaption() and updateAltText() are wired to onValueChange and execute getFinalPostContent + saveFinalPostContent synchronously on the main thread, while every other operation in the screen goes through operationMutex + Dispatchers.IO. This risks jank and StrictMode violations and creates an unsynchronized read-modify-write path that can race the IO-serialized operations.
- Hashtag prefix mismatch between preview and shared text (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/FinalPostComposerViewModel.kt): The screen normalizes bare hashtag strings by prepending # when missing, but shareOrExport() joins state.hashtags raw. If stored generated hashtags omit the # prefix, the text shared to external apps differs from the in-app preview.
- Workspace navigates to composer before async ReadyToShare transition completes (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/ManualPostDraftWorkspaceScreen.kt): onShareOrExport fires markShareOrExportStarted() asynchronously and then calls onNavigateToFinalComposer() immediately. If the guard inside markShareOrExportStarted rejects (no caption, already Shared), its error message is never visible and the composer opens anyway. Additionally the workspace save snapshot taken before navigation can race a concurrent composer-side write.
- effectiveShortCaption helper added to domain model but never used (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/FinalPostComposerViewModel.kt): The ViewModel reads latestCaptionResult?.shortCaption directly instead of through effectiveShortCaption(), contradicting the helper's semantics that a manual caption hides the short caption. A test asserts shortCaption == "Generated short" alongside a manual caption, enshrining the inconsistency.
- No path to reset editedCaption to null once set (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/FinalPostComposerViewModel.kt): effectiveCaption() only checks editedCaption != null and returns it. Once the user manually edits the caption there is no UI affordance to revert to the generated caption, and saving an empty string as editedCaption blocks reversion.
- App.kt fallback shows misleading error when only share/clipboard dependencies are null (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/App.kt): The "Draft repository not available" message fires even when the draft repository exists and only clipboardWriter or postShareLauncher are null, which would misrepresent the failure to users on future non-Android targets.

## Disputed Issues

- Severity of unguarded transitionTo crash: Claude rated it blocking (transitionTo is a require that throws IllegalArgumentException for Draft→ReadyToShare and Archived→anything, and the composer receives no status guarantee from the workspace); Codex initially mischaracterised the composer as tolerant of earlier statuses but corrected this in critique round, treating it as a real robustness defect with narrow normal-UI reachability. ## Outstanding Concerns

- Unguarded transitionTo in shareOrExport: the workspace can navigate to the composer while the draft is in Draft or Archived status (stale state, programmatic navigation, or a slow markShareOrExportStarted), and the composer calls draft.transitionTo(ReadyToShare) without a canTransitionTo guard, which throws IllegalArgumentException inside a rememberCoroutineScope-launched coroutine and crashes the app. The guard present in ManualPostDraftWorkspaceViewModel.kt:417 is absent here.
- Android share semantics are now settled for this branch: `Exported` and `Shared` mean the post was handed off to the Android OS chooser, not that an external target-app share completed. The remaining concern is documentation and UI wording, not the persisted workflow definition itself.
