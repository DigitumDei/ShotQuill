## Blocking Issues

- Latest commit not pushed to origin (docs/reviews/ask-816-1781066002893/2026-06-10T09-46-34.213Z-review.md): Local HEAD is 798f370 ('Clarify chooser-handoff review guidance') while origin/ask/816-1781066002893 is at 710ccf5 — the branch is ahead by one. The explicit success criterion for this turn (push the review artifact commit to origin) is unmet. Critically, the unpushed commit rewrites the review consensus record, reclassifying the 'share launch incorrectly treated as completed sharing' finding from a blocker to accepted chooser-handoff policy, so the version visible on origin is materially different from the current authoritative local record.
- Per-keystroke synchronous SQLite I/O on the UI thread (app/src/main/kotlin/…/FinalPostComposerViewModel.kt): FinalPostComposerViewModel.updateCaption and updateAltText are called directly from Compose onValueChange callbacks, and each synchronously calls repository.getFinalPostContent then repository.saveFinalPostContent on the main thread. Every other repository call in this VM is routed through Dispatchers.IO behind operationMutex; these two bypass both, creating jank/ANR risk on the hottest input path and a state-clobber race where a keystroke's state.copy can be silently overwritten by a concurrently completing load or selectOriginalPhoto on the IO mutex.
- Inconsistent canShare gate allows sharing empty caption after reload or photo switch (app/src/main/kotlin/…/FinalPostComposerViewModel.kt): updateCaption computes canShare = text.isNotBlank() && …, but toState() computes canShare = effectiveCaption(content) != null && …. When the user clears the caption, updateCaption persists editedCaption = "" (a non-null override). Any subsequent action that rebuilds state from the draft (selectOriginalPhoto, selectEditedPhoto, a successful share, or reopening the screen) calls toState(), which treats the non-null empty string as a valid caption and sets canShare = true. shareOrExport then composes state.caption ?: "", sharing an empty caption despite the validation message claiming otherwise.

## Non-Blocking Issues

- Shared hashtags not '#'-normalized in share payload (app/src/main/kotlin/…/FinalPostComposerViewModel.kt): The UI defensively renders bare tags with a '#' prefix, but shareOrExport builds the share text with state.hashtags.joinToString(" ") without normalizing. PostTextGenerationPipeline trims but does not prefix, so bare tags are possible and the shared text would differ from what the user saw. Existing tests only use pre-prefixed '#test' so this is untested.
- Missing transition guard in shareOrExport (app/src/main/kotlin/…/FinalPostComposerViewModel.kt): The workspace VM checks draft.canUpdateOrTransitionTo(DraftStatus.ReadyToShare) before transitioning; the composer VM calls draft.transitionTo(DraftStatus.ReadyToShare, ...) unguarded. An illegal transition throws IllegalArgumentException from inside coroutineScope.launch, crashing the app. Currently latent — only legal statuses are reachable in normal navigation — but a defensive asymmetry that will surface the first time an Archived or future-status draft reaches the screen.
- markShareOrExportStarted is redundant and effectively best-effort: The workspace fires refresh { markShareOrExportStarted() } and then navigates in the same callback. Navigation removes the workspace from composition and cancels its rememberCoroutineScope, so the ReadyToShare transition may not complete. The composer repeats the transition itself in shareOrExport, making the workspace call redundant and a potential write race against the composer's load (two separate mutexes, same draft row).
- effectiveShortCaption is dead code contradicting actual behavior (app/src/main/kotlin/…/ManualWorkflowModels.kt): effectiveShortCaption is defined in ManualWorkflowModels.kt (returns null when a manual caption override exists) but never called; toState() reads latestCaptionResult?.shortCaption directly. The load test asserts the generated short caption survives a manual override, silently contradicting the function's semantics. The function should be used or deleted.
- Misleading fallback message in App.kt for missing dependencies (app/src/main/kotlin/…/App.kt): The FinalComposer route displays 'Draft repository not available' even when the actual missing dependency is clipboardWriter or postShareLauncher. Currently harmless on Android targets, but will mislead on the first non-Android target or in test previews.
- Failed share leaves UI state stale after repository mutation (app/src/main/kotlin/…/FinalPostComposerViewModel.kt): On a share failure, only statusMessage is updated in state, but the repository draft was already mutated (possibly transitioned to ReadyToShare plus a Failed export record written). The success path refreshes from the repository; the failure path should do the same to keep state consistent with persisted data.
- removePrefix("file://") URI handling fragile with percent-encoded paths: Percent-encoded paths (spaces as %20, etc.) will fail the exists() check and silently degrade to 'share failed'. Likely unreachable with app-generated paths, but worth noting since the launcher accepts arbitrary stored URIs.
