## Executive Summary

This branch implements a post-text generation pipeline for ShotQuill that orchestrates media vision analysis, prompt assembly, AI provider calls, and persistence of caption results including a new optional `shortCaption` field, along with prompt history tracking and regeneration support. The central and unambiguous blocking finding — confirmed by all three reviewers — is that `PostTextGenerationPipeline` is never invoked by `ManualPostDraftWorkspaceViewModel`: the user-facing generate action still routes through `ManualDraftAiProvider` (defaulting to `FakeManualDraftAiProvider`), so the feature does not execute in production at all. A second structural problem compounds this: the pipeline's read-modify-write aggregate save pattern, combined with `SqlDelightManualWorkflowRepository.save()` deleting all owned child rows before reinserting from a potentially stale in-memory snapshot, can silently overwrite newer draft state or truncate accumulated history. These two issues, taken together, mean the branch adds infrastructure that is both unreachable and unsafe to wire in without additional fixes, making `revise` the appropriate verdict.

**2. `shortCaption` type contract is implicit and fragile.**
`CaptionGenerationOutput.shortCaption` is `String` (non-nullable), but the pipeline stores it as nullable via `.trim().takeIf { it.isNotEmpty() }`. The interpretation "empty string means no short caption" is an implicit convention in the pipeline, not in the provider contract. If a future provider returns `" "` intending to signal absence, the `.trim().takeIf` handles it correctly. But the contract should live at the interface boundary (`CaptionGenerationOutput.shortCaption: String?`) instead of being silently normalized downstream. This is a latent API design issue.

**3. Redundant null-safety in `responseSummary()`.**
```kotlin
shortCaption?.takeIf { it.isNotBlank() }?.let { ... }
```
`shortCaption` is already guaranteed to be non-blank by construction (it was stored as null if blank, or as a trimmed non-empty string). The `isNotBlank()` check is dead code. It signals unclear thinking about what invariants are already guaranteed — either the check is wrong (unnecessary) or the construction is wrong (should allow blanks). Currently it's just misleading.

**4. TOCTOU between the two `repository.get(draftId)` calls.**
The pipeline reads the draft once for status validation, calls vision analysis (which performs its own read-modify-write), then reads the draft again as `currentDraft`. If another coroutine modifies the draft between the second read and `repository.save(updatedDraft)`, the save silently overwrites those changes (last-write-wins on the full draft). For a single-user mobile app this is probably acceptable, but it contradicts the thread-safety framing of the PR. Worth a comment at minimum.

A public class with a private `internal` counter type is fine, but it means the atomicity guarantee is invisible to callers. Not a bug, but worth noting for future platform ports.
