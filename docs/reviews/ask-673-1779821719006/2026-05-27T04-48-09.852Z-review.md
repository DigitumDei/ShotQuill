## Executive Summary

This branch implements a 'New Post' feature for the ShotQuill Android app, adding a 3-step UI (source selection, processing spinner, success/error) with camera capture and gallery import via PickVisualMedia, file copying to filesDir/media/originals/, metadata extraction, and PostDraft/MediaAsset persistence. All three reviewers converged on two severe defects: the 'Try Again' recovery path after a draft-save failure is permanently broken because onDismissError clears captureError/saveError but not captureResult, leaving LaunchedEffect(captureResultValue) with an unchanged key so the spinner runs forever with no escape; and MediaFileManager — the PR's most side-effect-heavy class — is entirely untested and explicitly excluded from the 90% Kover threshold, directly circumventing the stated success criteria. A further cluster of non-blocking defects (non-atomic repository saves risking orphaned MediaAsset rows, cancellation incorrectly routing to the error step, state loss on configuration changes, raw filesystem paths stored in a field named uri, spinner appearing only after file copy completes rather than during it) adds technical debt but does not individually block correctness. The unrecoverable stuck-state regression and the deliberate coverage exclusion for the riskiest code together justify a revise verdict.

## Blocking Issues

- 'Try Again' after save failure leads to permanently stuck spinner (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/App.kt): onDismissError (App.kt:129-132) clears captureError and saveError but not captureResult. After dismissal: captureResult != null, draftCreatedMessage == null, errorMessage == null → NewPostStep.Processing. LaunchedEffect(captureResultValue) is keyed on the same unchanged object reference and does not re-fire. The spinner runs forever. There is no user-reachable recovery path short of killing the app.
- MediaFileManager fully excluded from Kover, bypassing the 90% coverage threshold (build.gradle.kts): build.gradle.kts lines 23-30 add com.digitumdei.shotquill.media.*, AppKt, AppScreen, NewPostScreenKt, and MainActivity* to the Kover exclusion list. The success criteria explicitly require a 90% CI coverage threshold, but this exclusion targets exactly the new code with the most complex, failure-prone logic (directory creation, stream copy, BitmapFactory decoding, MIME-type guessing, Random-based file naming). Zero tests exist for MediaFileManager, and there are no instrumented tests either.

## Non-Blocking Issues

- NewPostCreator.createDraftFromMedia is non-atomic — orphaned MediaAsset on draft save failure (shared/src/commonMain/kotlin/com/digitumdei/shotquill/workflow/NewPostCreator.kt): mediaAssetRepository.save(mediaAsset) at NewPostCreator.kt:42 persists before postDraftRepository.save(postDraft) at line 63, with no rollback or transaction wrapper. If the draft save throws, a MediaAsset row exists with no referencing PostDraft. SqlDelightManualWorkflowRepository implements both repository interfaces and could expose a single transactional save method.
- Cancellation (camera back-press, gallery back-press) incorrectly routes to the error state (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/media/MediaCaptureHandler.kt): MediaCaptureHandler.kt:62 calls onError('Camera capture cancelled') on a false result, and line 98 calls onError('Gallery selection cancelled') on a null URI. Both are normal user-initiated exits; they should silently return to ChooseSource rather than pushing the user to NewPostStep.Error and requiring an explicit 'Try Again' tap.
- captureResult and captureError use remember, not rememberSaveable — state lost on configuration change (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/MainActivity.kt): remember { mutableStateOf(...) } in MainActivity.kt:32-33 does not survive Activity recreation (rotation, locale change). cameraFilePath correctly uses rememberSaveable, creating an asymmetry: the saved path points to a valid file after recreation, but the result and error state it relates to are gone, silently returning the user to ChooseSource.
- URI field stores absolute filesystem paths, not URI-formatted values (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/media/MediaFileManager.kt): MediaFileManager returns destFile.absolutePath / captureFile.absolutePath into MediaCaptureResult.uri. Tests in MediaCaptureResultTest.kt use bare paths while NewPostCreatorTest.kt uses file:// URIs. Any downstream caller using Uri.parse() on this field will silently produce a relative-path URI rather than a file:// URI, which is a latent crash risk.
- Spinner does not cover the file-copy and metadata-extraction phase (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/media/MediaCaptureHandler.kt): MediaCaptureHandler performs handleCameraCapture / importFromContentUri (the heaviest IO: stream copy, BitmapFactory dimension extraction) inside its coroutine scope before calling onResult. captureResult is set by onResult, so the UI only reaches NewPostStep.Processing after the IO work is already done. The spinner is shown after the latency the user would most notice.
- Step-derivation logic is duplicated in tests rather than shared with production (composeApp/src/androidUnitTest/kotlin/com/digitumdei/shotquill/MediaCaptureResultTest.kt): The private deriveStep() helper in MediaCaptureResultTest.kt manually replicates the when expression from NewPostScreen.kt:49-55. If the production priority ordering changes (e.g., Error before Complete), the duplicated test helper silently continues passing without exercising the real code path.
- Timestamps reflect processing time, not capture time; MediaCaptureResult.createdAtEpochMillis is discarded (shared/src/commonMain/kotlin/com/digitumdei/shotquill/workflow/NewPostCreator.kt): MediaFileManager calls System.currentTimeMillis() after IO completes in both handleCameraCapture (line 53) and importFromContentUri (line 19), recording copy time rather than capture time. More critically, NewPostCreator.createDraftFromMedia ignores the createdAtEpochMillis field on MediaCaptureResult entirely and calls clock.nowMillis() at line 30, so the persisted MediaAsset.createdAtEpochMillis is the draft-creation time, not even the file-copy time.

```text
---

## Blocking Issues

**1. "Try Again" leads to an unrecoverable stuck state.**

In `App.kt`, `onDismissError` calls `onClearCaptureError?.invoke()` (sets `captureError = null` in MainActivity) and `saveError = null`. Neither path clears `captureResult`. After the dismiss: `captureResult != null`, `draftCreatedMessage == null`, `errorMessage == null` → step derives to `Processing`. But `LaunchedEffect(captureResultValue)` does **not** re-fire because the key — the same `captureResult` object reference — has not changed. The spinner runs forever. The user cannot retry after a draft-save failure. The "Try Again" button is broken for the `saveError` path.

**2. `MediaFileManager` has zero tests and is explicitly excluded from the coverage threshold.**

`build.gradle.kts` adds `"com.digitumdei.shotquill.media.*"` to the Kover exclusion list. This is the most dangerous file in the PR: it creates directories, copies streams, calls `BitmapFactory`, guesses MIME types, and uses `Random` for file naming — all with real side effects and real failure modes. The 90% CI threshold in the success criteria is neutralised for this class before a single test is written. There is no instrumented test either.

---

## Non-Blocking Issues

**1. State does not survive configuration changes.**

`captureResult` and `captureError` in `setContent {}` are plain `remember { mutableStateOf(...) }`. Rotating the device during Processing silently drops the ongoing capture state and returns the user to ChooseSource with no indication of what happened. `cameraFilePath` in `rememberMediaCaptureHandler` correctly uses `rememberSaveable`, but the result state one level up does not.

**2. Cancellation is treated as an error.**

Both camera back-press (`success == false`) and gallery picker back-press (`uri == null`) call `onError(...)`, which routes the user to `NewPostStep.Error` and shows an error-styled message requiring an explicit "Try Again" tap. Cancellation is a normal user action. It should silently return to `ChooseSource`, not produce an error state.

**3. `sqldelight.android.driver` added directly to `composeApp`.**

`composeApp/build.gradle.kts` adds `libs.sqldelight.android.driver` to `androidMain`. The `shared` module already carries this dependency and exposes `AndroidDatabaseDriverFactory`. Having the UI module import the driver directly bypasses the module boundary and suggests a build error was masked rather than understood.

**4. `handleCameraCapture` timestamps processing time, not capture time.**

`System.currentTimeMillis()` is called inside `handleCameraCapture`, which runs after the coroutine is dispatched to IO. The file was written by the camera app seconds (or more) earlier. `createdAtEpochMillis` on the `MediaAsset` ends up reflecting when the app processed the file, not when the photo was taken. `captureFile.lastModified()` would be more accurate. The same issue exists in `importFromContentUri`, where `now` is the copy time, not the photo's creation time. Both are compounded by `NewPostCreator.createDraftFromMedia` ignoring the timestamp passed in and calling `clock.nowMillis()` again — so the timestamps from `MediaCaptureResult` are not what ends up in the database.

**5. `NewPostCreator` is non-atomic — orphaned media asset on draft save failure.**

`mediaAssetRepository.save(mediaAsset)` persists before `postDraftRepository.save(postDraft)`. If the draft save throws, there is a saved `MediaAsset` with no referencing `PostDraft`. No cleanup, no transaction wrapper. The repositories share a `ManualWorkflowRepository` type; a transactional save method could prevent the split.

**6. `remember(context, mediaFileManager)` key is insufficient in `rememberMediaCaptureHandler`.**

The returned `MediaCaptureHandler` captures `cameraPermissionLauncher`, `cameraLauncher`, and `galleryLauncher` by closure. These are only recreated when the composable leaves and re-enters composition. If the permission result triggers a hot-path recomposition that invalidates the launcher references before the `remember` key changes (e.g., unusual lifecycle events), the handler holds stale launchers. In practice this is unlikely, but the correctness argument depends on Compose internals rather than explicit key dependencies.

---

## Missing Tests

- **`MediaFileManager` — completely untested.** File name generation, directory creation, MIME-type guessing (`"image/heic"` vs `"image/heif"` overlap in `mimeTypeToExtension`), stream copy, zero-byte file handling, `readImageDimensions` with corrupt/empty files — none of it is exercised. - **`captureResult != null` with `newPostCreator == null`** — no test for the stuck-spinner scenario where the repository is not provided but a capture result arrives.
- **Cancellation behaviour** — no test asserting that back-press from camera or gallery returns to `ChooseSource`. The current behaviour (error state) is likely wrong.
- **`MediaCaptureHandler` coroutine paths** — the `CancellationException` re-throw, the file delete on cancel, and the IO coroutine for both camera and gallery are all untested.

---
```
