## Executive Summary

This branch introduces camera and gallery media capture for new post creation and addresses a series of prior adversarial review findings covering race conditions, thread safety, file cleanup, and ID uniqueness. Three blocking issues remain unresolved: `MediaCaptureResultSaver.restore` crashes with `IndexOutOfBoundsException` on any input with fewer than four pipe-delimited segments (including a test case that claims to verify graceful handling of exactly this input), `File.delete()` is called synchronously on the main thread in two distinct locations, violating the explicitly stated off-main-thread I/O criterion, and the camera capture file is permanently orphaned when `handleCameraCapture` throws because `cameraFilePath` is nulled before the coroutine runs, `onResult` never fires, and the `.onFailure` branch only calls `onError` with no file deletion. A fourth proposed blocking issue — that `NewPostCreator` fails to persist `MediaAsset` to the database — was raised by one reviewer and subsequently self-retracted after direct inspection confirmed that `SqlDelightManualWorkflowRepository.save(postDraft)` cascades media asset saves transactionally. The three confirmed blocking issues are verifiable by direct code reading with no ambiguity and must be resolved before the branch is ready.

## Blocking Issues

- `MediaCaptureResultSaver.restore` crashes on truncated saved state (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/model/MediaCaptureResult.kt): Lines 23-26 of MediaCaptureResult.kt access `parts[1]`, `parts[2]`, and `parts[3]` with direct array indexing after `split("|", limit = 5)`. Any serialized string with fewer than four pipe characters — including the string `"file:///only/uri.jpg"` passed in `saverRestoreHandlesTruncatedInputGracefully` — throws `IndexOutOfBoundsException`. The test named to verify graceful truncation handling will fail, not pass. OS-truncated saved state from process death mid-write will crash the app on restore. Only `parts[4]` uses safe `getOrNull` access.
- Blocking `File.delete()` calls on the main thread (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/media/MediaCaptureHandler.kt): `MainActivity.kt:64` calls `File(path).delete()` synchronously inside `onCleanupCapture`, which is invoked from an `onClick` handler on the main thread. `MediaCaptureHandler.kt:65` calls `runCatching { File(path).delete() }` inside the `cameraLauncher` activity-result callback, also on the main thread. Both violate the stated success criterion that all I/O runs off the main thread. The inconsistency is underscored by the correctly dispatched `previousPath` deletion at line 85, which uses `withContext(Dispatchers.IO)`.
- Camera capture file permanently orphaned when `handleCameraCapture` throws (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/media/MediaCaptureHandler.kt): In `MediaCaptureHandler.kt`, `cameraFilePath` is set to null before the processing coroutine begins. If `mediaFileManager.handleCameraCapture(File(path))` subsequently throws, the `.onFailure` branch calls `onError` but does not delete `File(path)`. Because `onResult` was never called, `captureResult` remains null in `App.kt`. The `onDismissError` guard `if (captureResult != null)` is false, so `onCleanupCapture` is never triggered. The file at `path` accumulates in `filesDir/media/originals/camera/` with no recovery path.

## Non-Blocking Issues

- Pipe-delimiter saver format corrupts URIs containing `|` (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/model/MediaCaptureResult.kt): The `|` separator in `MediaCaptureResultSaver` is used without escaping. The test `saverRestoreHandlesPipeInUriField` only asserts `assertNotNull(restored)` and does not verify that the URI field is preserved intact. For a URI containing `|`, `parts[0]` would be truncated at the first pipe. App-generated `file://` URIs are unlikely to contain pipes in practice, making this low probability but fragile by design.
- Uniqueness test asserts only that two of ten names differ (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/media/MediaFileManagerTest.kt): `MediaFileManagerTest` asserts `names.size > 1` across 10 generated names, which passes if even one pair differs. This does not verify the uniqueness guarantee the production code intends. The assertion should verify `names.size == 10` (all distinct).
- Shared mutable `cameraFilePath` state yields wrong file on rapid double-tap (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/media/MediaCaptureHandler.kt): On rapid double-tap, a second permission callback can overwrite `cameraFilePath` while the first camera callback is still in flight. The first callback then reads and deletes the second launch's file path from the now-mutated shared state, causing the second launch to lose its capture target. Each launch should capture its own path in a closure rather than reading shared mutable state.
- Uncaught exceptions from `createCameraCaptureFile` and `FileProvider.getUriForFile` do not call `onError` (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/media/MediaCaptureHandler.kt): Failures in file creation or URI generation inside the permission-callback coroutine are not caught, so `onError` is never called and the UI is left unaware of the error state. The spinner or loading state would be stuck.
- `queryDisplayName` silently swallows all exceptions including unexpected ones (composeApp/src/androidMain/kotlin/com/digitumdei/shotquill/media/ContentResolverMediaImporter.kt): The broad `catch (_: Exception)` in `ContentResolverMediaImporter.kt` returns null for any failure, including programming errors. This silently degrades to a generated filename with no signal that something unexpected occurred.
- `NewPostCreator` calls `clock.nowMillis()` twice with potentially different values (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/post/NewPostCreator.kt): The default parameter `createdAtEpochMillis: Long = clock.nowMillis()` and the body call `val nowEpoch = clock.nowMillis()` are two distinct invocations. With a real clock they could differ by microseconds. In production `createdAtEpochMillis` is always passed explicitly so the default never fires, but the default-parameter semantics are misleading and the two reads remain distinct.
