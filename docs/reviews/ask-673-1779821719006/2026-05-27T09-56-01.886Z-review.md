## Executive Summary

This branch implements a new post creation flow (Issue 7) allowing users to create posts from camera captures or gallery imports, spanning a shared domain layer (NewPostCreator, MediaAsset, PostDraft), a three-step Compose UI (NewPostScreen: source selection → processing → complete/error), and Android-specific handlers (MediaCaptureHandler, MediaFileManager, ContentResolverMediaImporter) storing media under filesDir/media/originals/. All four reviewers across both rounds unanimously identified two data-correctness bugs that must be fixed before merge: duplicate drafts written to SQLite on Activity recreation (captureResult survives via rememberSaveable while draftCreatedMessage resets to null, causing LaunchedEffect to re-fire with fresh random IDs), and a CancellationException swallowed by a bare catch (e: Exception) block in App.kt that sets a bogus error message and breaks structured concurrency. The verdict is revise because these blocking issues have no mitigation in the current code and no tests covering them.

## Blocking Issues

- Duplicate drafts written to SQLite on Activity recreation (app/src/main/java/com/digitumdei/shotquill/App.kt): captureResult is stored with rememberSaveable (MainActivity.kt:32) and survives rotation or process death, but draftCreatedMessage uses plain remember (App.kt:84) and resets to null. On recreation, LaunchedEffect(captureResultValue) fires on first composition, finds captureResultValue non-null, generates a fresh Random.nextInt suffix (App.kt:91–93), and writes a second PostDraft row referencing the same media file. There is no idempotency guard anywhere in the path.
- CancellationException swallowed in App.kt LaunchedEffect (app/src/main/java/com/digitumdei/shotquill/App.kt): catch (e: Exception) at App.kt:108 catches CancellationException (a subclass of RuntimeException → Exception in the Kotlin JVM hierarchy). If the coroutine is cancelled after createDraftFromMedia commits to SQLite but before withContext returns, the catch fires, sets saveError = "Failed to create draft: null" (CancellationException.message is typically null), and the coroutine completes normally instead of propagating the cancellation signal. MediaCaptureHandler correctly rethrows CancellationException at lines 52 and 99; App.kt does not.

## Non-Blocking Issues

- MediaCaptureResultSaver.restore crashes on malformed saved state (app/src/main/java/com/digitumdei/shotquill/MediaCaptureResult.kt): parts[4].toLong() at MediaCaptureResult.kt:27 throws IndexOutOfBoundsException if the saved string has fewer than five pipe-delimited segments, and NumberFormatException if the timestamp field is non-numeric. Width/height fields use toIntOrNull() and degrade gracefully; the timestamp does not. Reachable via saved-state corruption or an app-version format mismatch.
- Gallery-imported file orphaned on draft persistence failure (app/src/main/java/com/digitumdei/shotquill/ContentResolverMediaImporter.kt): ContentResolverMediaImporter copies the gallery image to filesDir/media/originals/gallery/ before returning. The catch block at ContentResolverMediaImporter.kt:37–40 deletes destFile only if the copy itself fails. If the copy succeeds but newPostCreator.createDraftFromMedia subsequently throws (disk full, SQLite error, coroutine cancellation), the copied file persists on disk with no corresponding database row and is never cleaned up.
- Processing spinner appears after I/O completes, not during (app/src/main/java/com/digitumdei/shotquill/NewPostScreen.kt): handleCameraCapture and importFromContentUri run on Dispatchers.IO and set captureResult only on completion (MediaCaptureHandler.kt:47–50, 90–97). deriveNewPostStep returns Processing only when captureResult != null. - "Try Again" discards captured image on transient save failure (app/src/main/java/com/digitumdei/shotquill/App.kt): onDismissError at App.kt:132–136 unconditionally calls onClearCaptureResult, setting captureResult = null in MainActivity. If the error originated from draft persistence (saveError is set, captureError is null), the user's captured or imported image is lost and they must recapture. The button label "Try Again" implies retrying the write, not discarding the capture and starting over.

## Disputed Issues

- Pipe character in file URI corrupting MediaCaptureResultSaver: the split/toLong crash is real, but reviewers disagree on whether a pipe can appear in a generated URI in this codebase (all camera/gallery destination filenames are internally generated with safe characters, making this theoretical rather than a demonstrated execution path)
- Whether making draftCreatedMessage rememberSaveable alone fixes the duplicate-draft bug: consensus is that it does not — the correct fix requires either clearing captureResult immediately after successful save or adding an idempotency guard keyed on the media URI, not just persisting the completion flag

## Outstanding Concerns

- The CI coverage gate provides a false-positive quality signal: the 90% threshold is met by excluding all new functionality from measurement. A reviewer reading CI green cannot infer that the feature code is tested — the two blocking bugs are in excluded files.
- NewPostCreatorTest's fake PostDraftRepository manually cascades media assets inside save(PostDraft), coincidentally mirroring the real SqlDelightManualWorkflowRepository behavior. Nothing documents or enforces this correspondence; a future refactor breaking the real cascade would pass all NewPostCreator unit tests while silently breaking production.
- createCameraCaptureFile() calls dir.mkdirs() synchronously inside the cameraPermissionLauncher result callback, which runs on the main thread. This is a formal StrictMode violation on a blocking filesystem operation, absent from both reviewers' primary finding lists.

When null, picking an image calls `error("Gallery import not available")`, routing the user to the Error step.
