## Executive Summary

This branch decouples AI photo-edit orchestration from the ViewModel into a composable `PhotoEditExecutionPipeline`, adds `selected_media_asset_id` tracking to post drafts, and aligns the database schema via a new migration. The single confirmed blocking issue is a hard compile failure: commit `1e8451b` moved `selected_media_asset_id` to the last column of `post_drafts` in `ShotQuill.sq` but left the positional `SELECT *` mapper lambda in `SqlDelightManualWorkflowRepository.kt:155` unchanged, producing two Kotlin type-mismatch errors against `DraftRow`. Beyond the compile blocker, the branch's most significant design weakness is an error-taxonomy problem: 'draft missing,' 'source media asset not found,' and 'asset-not-owned-by-draft' all collapse into `DraftNotFound`/`unloadedState`, causing the ViewModel to destructively wipe workspace state on benign validation failures and surface misleading user messages. The branch must be revised before merge to fix the mapper order; the error-taxonomy issues are high-priority follow-up design feedback.

## Blocking Issues

- SQLDelight SELECT * mapper column order mismatch causes compile failure (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/storage/SqlDelightManualWorkflowRepository.kt): Commit 1e8451b reordered `selected_media_asset_id` to the last (8th) position in the `post_drafts` CREATE TABLE, but the hand-written positional lambda at line 155 still binds it at position 6. This feeds a `Long` (created_at_epoch_millis) into `DraftRow.selectedMediaAssetId: String?` and a `String?` (selected_media_asset_id) into `DraftRow.updatedAt: Long` — two irreconcilable type-mismatch compile errors. Fix: reorder the lambda to `..., brandProfileId, createdAt, updatedAt, selectedMediaAssetId` to match the current table declaration, or replace `SELECT *` with an explicit column list.

## Non-Blocking Issues

- DraftNotFound returned when the source media asset is missing, not the draft itself (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/ai/photoedit/PhotoEditExecutionPipeline.kt): `PhotoEditExecutionPipeline` successfully loads the draft before returning `PhotoEditExecutionError.DraftNotFound` when no media asset matches the vision description's `mediaAssetId`. The draft exists; the failure is a missing or mismatched source asset. This misclassification surfaces in the ViewModel as 'Draft not found' and permanently disables `canEditPhotoWithAi`, preventing recovery even when the draft is intact. A dedicated error variant (e.g. `SourceMediaAssetNotFound`) would produce an accurate message and preserve recoverability.
- updateSelectedMediaAsset false collapses two distinct failures into a destructive workspace wipe (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/ui/workspace/ManualPostDraftWorkspaceViewModel.kt): `SqlDelightManualWorkflowRepository` returns `false` from `updateSelectedMediaAsset` for both a missing draft and an asset that does not belong to the draft. The ViewModel maps both outcomes to `unloadedState("Draft not found")`, wiping the entire workspace on what may be a benign asset-validation failure. A non-destructive, correctly-labelled error path would avoid false data loss for the user.
- maskRegion is modeled and persisted but always written as null — dead plumbing (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/ai/photoedit/PhotoEditExecutionPipeline.kt): `PhotoEditExecutionPipeline` passes `maskRegion = null` at all four call sites. The domain object, persistence column, and prompt-assembler all support the field, but the execution API provides no path to populate it. If scoped/object-region edits are intentionally deferred, this should be noted explicitly so future contributors know the wiring is incomplete rather than broken.
- verifyMigrations enabled in build config but no committed schema snapshot exists (shared/build.gradle.kts): `shared/build.gradle.kts` configures `verifyMigrations {}` and `schemaOutputDirectory`, but only `1.sqm` and `ShotQuill.sq` exist — no `.db` baseline snapshot is committed. Without a baseline, Gradle-level migration verification cannot detect structural drift. The commit message 'Enable SqlDelight migration verification' overstates the protection; the real safety net is the hand-written `AndroidDatabaseDriverUpgradeTest`.
- Successful photo edit unconditionally overrides an explicit user media selection (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/storage/SqlDelightManualWorkflowRepository.kt): `savePhotoEditSuccess` always sets `selected_media_asset_id` to the newly edited asset. If a user deliberately chose 'Use original' before running another edit, that preference is silently discarded. This may be the intended UX, but it should be confirmed as a deliberate product decision rather than treated as an accidental side effect.
- catch (e: Exception) in editPhotoWithPipeline swallows CancellationException (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/ui/workspace/ManualPostDraftWorkspaceViewModel.kt): The broad `catch (e: Exception)` block in the ViewModel would intercept coroutine `CancellationException`. The photo-edit executor is currently synchronous so the risk is theoretical and the pattern is pre-existing, but the refactor that reorganised this block is the appropriate moment to add `catch (e: CancellationException) { throw e }` above the generic handler.

## Missing Tests

## Outstanding Concerns

- The error-taxonomy conflation is the highest-value design feedback beyond the compile fix: 'draft missing,' 'source media asset not found,' and 'asset-validation failure' all route through DraftNotFound/unloadedState, causing the ViewModel to destructively wipe workspace state and surface misleading messages. The author should address this holistically across PhotoEditExecutionPipeline and the ViewModel rather than as isolated nits.

After a successful edit the selected asset changes, the cache misses, and the pipeline correctly sources from the edited image. This behavior is load-bearing and subtle; a regression in the cache-key filter would silently make re-edits operate on the wrong image, and no test pins this specific interaction (selected-edited → re-edit sources from edited, not original).
