## Executive Summary

This branch implements an AI-powered photo editing pipeline for the ShotQuill Android app, including execution contracts, local file storage via AndroidPhotoEditMediaSaver, persistence of pipeline outcomes, and ViewModel state management for edited asset selection. The most critical defect is in PostDraft.primaryMediaAsset(): it searches only mediaItems for selectedMediaAssetId, but edited assets are stored in photoEditResults — causing every caller (pipeline re-entry, VisionDescriptionAnalyzer, PostTextGenerationPipeline) to silently fall back to the original photo, breaking the core requirement that the edited asset becomes the primary media. A second confirmed correctness failure is that persistFailure() never calls updateUpdatedAt(), violating the stated requirement that updatedAt advances on all draft mutations including failures. Compounding both bugs, the pipeline tests explicitly assert that mediaItems must remain unchanged after an edit, codifying the broken design rather than validating the intent. The branch is not ready to merge until these defects are corrected and the untested ViewModel selection APIs are covered.

## Blocking Issues

- primaryMediaAsset() cannot find edited photos (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/domain/ManualWorkflowModels.kt): PostDraft.primaryMediaAsset() searches only mediaItems for selectedMediaAssetId. The pipeline stores edited assets in photoEditResults and sets selectedMediaAssetId to the edited asset's ID, so every subsequent call to primaryMediaAsset() — including pipeline re-entry, VisionDescriptionAnalyzer.kt:36, and PostTextGenerationPipeline.kt:119,155 — silently falls back to the original photo. The existing domain tests mask this by constructing carousel drafts where the edited asset is manually inserted into mediaItems, a path never reached in the single-image edit flow.
- persistFailure() does not advance updatedAt (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/workflow/PhotoEditExecutionPipeline.kt): The persistFailure() path saves the edit request and prompt history entry to the repository but never calls updateUpdatedAt(). The returned updatedDraft retains the pre-operation updatedAt timestamp. The success criterion explicitly requires updatedAt to advance on every draft mutation, including failed edits. No test currently catches this because no test asserts the timestamp after a failure.
- selectEditedPhoto() and selectOriginalPhoto() have zero test coverage (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/ManualPostDraftWorkspaceViewModel.kt): These are new public ViewModel APIs with no tests anywhere in the suite. No test calls selectEditedPhoto() or selectOriginalPhoto(), and no assertion checks activePhotoUri, canSelectEditedPhoto, or whether selectedMediaAssetId is persisted after invocation. This directly contradicts the success criterion that active media selection is tracked and persisted.
- Pipeline tests codify the broken primaryMediaAsset() behavior (shared/src/commonTest/kotlin/com/digitumdei/shotquill/shared/workflow/PhotoEditExecutionPipelineTest.kt): PhotoEditExecutionPipelineTest.kt at lines 95, 224, 340, and 386 explicitly assert 'Original mediaItems must not be modified.' These assertions validate the current incorrect design and will actively resist a correct fix that adds edited assets to mediaItems or updates primaryMediaAsset() to search photoEditResults. The tests must be revised alongside the fix.

## Non-Blocking Issues

- prompt parameter is an opaque stub overwritten by the pipeline (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/ManualPostDraftWorkspaceViewModel.kt): The ViewModel constructs a terse prompt string ('Edit the image (intent, platform).') that the pipeline immediately discards in favour of PhotoEditPromptAssembler.assemble(editRequest). Callers cannot know their input is ignored. At minimum the parameter should be documented as a user-facing label rather than the assembled AI prompt, or removed if it serves no purpose.
- Selection UI callbacks are not wired from Screen to Content (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/ManualPostDraftWorkspaceScreen.kt): ManualPostDraftWorkspaceScreen.kt does not pass onSelectEditedPhoto or onSelectOriginalPhoto callbacks to ManualPostDraftWorkspaceContent, and the content component never reads or displays activePhotoUri or canSelectEditedPhoto. The ViewModel selection feature is fully implemented but unreachable from the UI.
- selectEditedPhoto() / selectOriginalPhoto() use full save() instead of updateSelectedMediaAsset() (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/ManualPostDraftWorkspaceViewModel.kt): The ViewModel calls postDraftRepository.save(updated) (full upsert) for both methods while updateSelectedMediaAsset() exists as a targeted interface method. Note: selectOriginalPhoto() must clear the selection to null, which updateSelectedMediaAsset() does not support, so a full save() may be intentional for symmetry — but the discrepancy should be documented or the interface method updated.

## Missing Tests

- primaryMediaAsset() with selectedMediaAssetId pointing to an asset that exists only in photoEditResults (not in mediaItems) — the exact single-image edit flow path
- selectEditedPhoto() selects the latest edited result, updates activePhotoUri, and persists selectedMediaAssetId through the repository
- selectOriginalPhoto() clears the selected media asset, restores the original URI, and persists the null selection

- Database migration from v1 to v2: create a v1 schema with pre-existing draft rows, run onUpgrade, verify selected_media_asset_id column exists and round-trip reads/writes succeed

## Disputed Issues

- Partial-write transactional integrity: OpenCode classifies fragmented individual writes (savePhotoEditRequest, savePhotoEditResult, updateStatus, etc.) without a wrapping transaction as blocking due to orphan-record risk if updateStatus() throws. Claude and Codex note the failure mode requires concurrent status mutation on a synchronous single-threaded pipeline, which is not a realistic scenario in the current threading model — they assess it as non-blocking technical debt rather than a shipping blocker.
- super.onUpgrade() omission in AndroidDatabaseDriverFactory: Codex flags the missing super.onUpgrade() call as bypassing SqlDelight's migration mechanism for future schema changes. OpenCode counters that SqlDelight's AndroidSqliteDriver.Callback.onUpgrade() default implementation drops and recreates all tables, so NOT calling super is the correct pattern for incremental migrations. Reviewers did not converge on the correct SqlDelight contract; the finding requires verification against the SqlDelight source before acting on it.

## Outstanding Concerns

- The primaryMediaAsset() fix has non-trivial design surface: the function must be extended to also search photoEditResults when selectedMediaAssetId is set. All callers (pipeline re-entry, VisionDescriptionAnalyzer, PostTextGenerationPipeline, toState()) depend on this function; the fix must be propagated and verified across all call sites, and the pipeline tests asserting mediaItems immutability must be revised to assert the correct post-edit primary asset instead.
- The full blast radius of the primaryMediaAsset() bug includes AltTextResult.mediaAssetId (PostTextGenerationPipeline.kt:155) pointing to the original photo rather than the edited one, meaning alt-text metadata is associated with the wrong asset ID in the database — this is a data integrity concern beyond display correctness.
