Several missing tests and one outstanding schema-behavior question also require attention before this is production-ready.

## Blocking Issues

- VisionDescriptionAnalyzer.imageSource.load() throws unguarded, bypassing failure-persistence (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/workflow/VisionDescriptionAnalyzer.kt): FileVisionImageSource.load() delegates to MediaFileManager.readMediaAssetBytes(), which throws IllegalArgumentException on a missing or stale file. This exception escapes PhotoEditExecutionPipeline.execute() entirely instead of producing a persisted FailedToLoadSourceImage record. The ViewModel's generic catch prevents a crash but swallows the event with no history entry, breaking the pipeline contract that all source-load failures are observable. FilePhotoEditImageSource.load() in the same codebase wraps the identical call in try/catch and returns SourceImageResult.Failure — the asymmetry is the defect. Fix: make VisionImageSource.load() return a typed Result, mirroring the photo-edit source pattern.
- maskRegion silently dropped at ViewModel call site — capability dead end-to-end (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/ManualPostDraftWorkspaceViewModel.kt): PhotoEditExecutor.execute() accepts maskRegion, the pipeline threads it to PhotoEditPromptAssembler and PhotoEditRequest, and the prompt assembler contains rich mask-region prose and tests. But ManualPostDraftWorkspaceViewModel.editPhotoWithPipeline() never passes a non-null value and ManualPostDraftWorkspaceScreen.kt has no UI control for it — the parameter always resolves to its default null. Either the feature is incomplete and the ViewModel must be finished, or maskRegion is future work and should be removed from the interface and pipeline until it is actually wired.

## Non-Blocking Issues

- Inconsistent error message prefix for the same class of provider failure (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/ManualPostDraftWorkspaceViewModel.kt): A bare Provider error (e.g. MissingApiKey surfacing from vision analysis) shows 'Unable to edit photo: …', while the same error wrapped in FailurePersisted from the editing step shows 'Photo edit failed: …'. Same underlying AiError, two different user-visible prefixes based on which persistence path was taken.
- Dead when branches for bare FailedToLoadSourceImage / FailedToSaveEditedImage in ViewModel (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/ManualPostDraftWorkspaceViewModel.kt): The concrete pipeline always wraps both error types in FailurePersisted before they reach the ViewModel, making the bare-error arms unreachable from the current implementation. They exist only for sealed-class exhaustiveness, implying a contract that does not hold and inviting future misunderstanding.
- FakeManualWorkflowRepository.savePhotoEditSuccess bypasses transitionTo(), reducing pipeline test fidelity (shared/src/commonTest/kotlin/com/digitumdei/shotquill/shared/workflow/PhotoEditExecutionPipelineTest.kt): The fake copies status directly without calling transitionTo(), bypassing the canTransitionTo invariant enforced by the real SqlDelightManualWorkflowRepository. Two pipeline instances starting within the same millisecond can generate the same filename. The destFile.exists() guard makes this a safe failure rather than data corruption, but the resulting error message would be misleading.
- Migration column order divergence — enable verifyMigrations as a robustness measure (shared/src/commonMain/sqldelight/migrations/1.sqm): 1.sqm appends selected_media_asset_id as the last physical column via ALTER TABLE while ShotQuill.sq declares it mid-table. Moving the new column to the end of CREATE TABLE and enabling verifyMigrations would make fresh-install and migrated schemas byte-identical and eliminate the entire class of risk.
