## Executive Summary

This branch adds persisted active-media selection to PostDraft, allowing users to toggle between the original and last-edited photo with the choice surviving process death. Three independent reviewers reached consensus on a critical domain model defect: `primaryMediaAsset()` resolves `selectedMediaAssetId` only from `mediaItems`, but the pipeline stores edited assets in `photoEditResults`, so selecting an edited photo has no effect on any pipeline operation (re-edit source, vision analysis, alt-text generation) — only the UI display path in `toState()` has the correct dual-collection fallback. The feature is also unreachable in practice because the screen composable never wires `selectEditedPhoto`/`selectOriginalPhoto` callbacks or renders `activePhotoUri`. Combined with the `persistFailure` timestamp gap and absent test coverage for every new public API, the branch is not ready to merge.

## Blocking Issues

- `primaryMediaAsset()` cannot resolve an edited-photo selection (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/domain/ManualWorkflowModels.kt): Lines 49–56 of ManualWorkflowModels.kt search only `mediaItems` for `selectedMediaAssetId`. After a successful edit the pipeline writes the edited asset's ID as the selection (PhotoEditExecutionPipeline.kt:232), but that asset lives in `photoEditResults`, not `mediaItems`. Every subsequent `primaryMediaAsset()` call silently falls back to the lowest-order original photo. This corrupts the source image for re-edits (pipeline line 104), the image analysed by VisionDescriptionAnalyzer (line 36 of that file), and the asset ID used for alt-text generation (PostTextGenerationPipeline.kt:119,155). The domain tests at ManualWorkflowModelsTest.kt:272–289 and 322–344 only pass because they place the edited asset inside `mediaItems` — the realistic single-photo edit scenario is never exercised.
- UI selection affordances are never wired in the screen composable (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/ManualPostDraftWorkspaceScreen.kt): ManualPostDraftWorkspaceScreen.kt (lines 96–111 and 155) passes no `onSelectEditedPhoto` or `onSelectOriginalPhoto` callbacks to `ManualPostDraftWorkspaceContent`, and the content composable never renders `activePhotoUri` or `canSelectEditedPhoto`. All three selection-related ViewModel methods and state fields are dead from the user-facing surface — the feature is inaccessible at runtime despite being fully implemented in the ViewModel and repository layers.
- `persistFailure` does not advance the draft's `updatedAt` timestamp (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/workflow/PhotoEditExecutionPipeline.kt): `persistFailure()` at PhotoEditExecutionPipeline.kt:133–158 inserts a `PhotoEditRequest` row and a `PromptHistoryEntry` row but never calls `repository.updateUpdatedAt`. The draft's `updated_at` column stays at its pre-failure value even though child rows were added. `FailurePersisted.updatedDraft` is then constructed with the stale `currentDraft.updatedAt` (consumed by the ViewModel at line 426-432), so any staleness or ordering check on in-memory state sees an unchanged timestamp. The success path correctly advances the timestamp; the failure path must too.

## Non-Blocking Issues

- `updateSelectedMediaAsset` interface cannot clear the selection to null (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/storage/ManualWorkflowRepositories.kt): The method signature takes a non-nullable `MediaAssetId`, so `selectOriginalPhoto()` cannot call it and instead falls back to a full `repository.save()` re-upsert. The interface contract is incomplete for the feature it supports: toggling selection should be a symmetric, narrow operation in both directions.
- `canSelectEditedPhoto` compares URIs instead of IDs (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/ManualPostDraftWorkspaceViewModel.kt): The flag evaluates `editedPhoto?.uri != activePhoto?.uri`. ID comparison is the semantically correct contract; URI comparison is fragile and would produce wrong results if two distinct assets share a path (e.g., test data or a file overwrite).
- `PhotoEditRequest.prompt` stores a caller-supplied debug string, not the assembled prompt (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/workflow/PhotoEditExecutionPipeline.kt): The ViewModel passes `"Edit the image (${intent}, ${platform})"` as `prompt`; the pipeline immediately uses `PhotoEditPromptAssembler.assemble(editRequest)` for the actual AI call and stores the real prompt only in `PromptHistoryEntry`. Any code inspecting `PhotoEditRequest.prompt` sees stale debug data. The parameter should either store the assembled prompt or be removed from the interface.
- No `canSelectOriginalPhoto` flag exposed from ViewModel state (composeApp/src/commonMain/kotlin/com/digitumdei/shotquill/screen/ManualPostDraftWorkspaceViewModel.kt): `canSelectEditedPhoto` is present, but the UI has no way to know when 'use original' is meaningful (only when an edited photo is currently active). Without this flag, the UI must reconstruct the logic from mutable URI fields — the same fragile comparison already flagged above.

## Outstanding Concerns

- The 6 sequential, non-transactional write calls in the pipeline success branch (PhotoEditExecutionPipeline.kt:222–232) create a real data-integrity window: a coroutine cancellation or crash after `savePhotoEditResult` but before `updateSelectedMediaAsset` leaves `selected_media_asset_id` unset with an orphaned result row. While the exact blocking severity is disputed, wrapping these in a single SQLDelight transaction would eliminate the risk entirely.
- The domain test suite gives false confidence on the core blocking issue: tests for `primaryMediaAsset()` + `selectedMediaAssetId` pass only because they use a non-representative setup (edited asset placed directly in `mediaItems`). The passing tests do not validate the production code path and could mask regressions after fixing the blocking issue.
