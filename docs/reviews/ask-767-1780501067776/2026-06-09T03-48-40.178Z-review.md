## Executive Summary

This branch introduces a dedicated `PhotoEditExecutor` pipeline for AI-powered photo editing, extracting the feature from the ViewModel's in-process `ManualDraftAiProvider` and mirroring the existing `PostTextGenerator` pattern. It also adds a `selected_media_asset_id` column via a v1→v2 schema migration with corresponding UI selection wiring. Two hard blockers prevent merging: `PhotoEditExecutionPipelineTest.kt` redeclares `val originalAsset` in the same function scope at lines 90 and 102, causing a Kotlin compile error that invalidates all test coverage claims; and no `.sqm` migration files exist under `shared/src/commonMain/sqldelight/`, while both the test suite assertion and the `AndroidDatabaseDriverFactory.onUpgrade` version gate rely on `Schema.version == 2` — meaning the upgrade path for existing v1 installs is almost certainly broken. The verdict is `revise` pending resolution of both blockers and the supporting test gaps.

## Blocking Issues

- Duplicate `val originalAsset` declaration causes compile error (shared/src/commonTest/kotlin/com/digitumdei/shotquill/shared/workflow/PhotoEditExecutionPipelineTest.kt): Two `val originalAsset` declarations appear at the same block level within a single test function in `PhotoEditExecutionPipelineTest.kt` (lines 90 and 102). Kotlin rejects this as a conflicting declaration, so `commonTest` will not compile and no pipeline tests can run.
- Missing `.sqm` migration files breaks schema v1→v2 upgrade path (shared/src/commonMain/sqldelight/ShotQuill.sq): No `.sqm` migration files exist under `shared/src/commonMain/sqldelight/` — only the full-schema `ShotQuill.sq`. SqlDelight derives `Schema.version` from migration files; with zero `.sqm` files the generated version is 1, not 2. This means `hasMigrationScaffoldForVersionTwo` will fail, and `AndroidDatabaseDriverFactory.onUpgrade` (which only calls `migrateV1ToV2` when `oldVersion < 2`) will never fire for existing v1 installs. Any device upgrading from the pre-column schema will lack `selected_media_asset_id`, causing `selectPostDraftById` to crash with 'no such column'. Fresh installs are unaffected since the column is in `CREATE TABLE`, which masks the defect in local testing.

## Non-Blocking Issues

- Null exception message yields `"Photo edit failed: null"` (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/workflow/ManualPostDraftWorkspaceViewModel.kt): `e.message` is nullable in the `editPhotoWithPipeline` catch block; a thrown exception with no message produces the literal string `"Photo edit failed: null"`. The replaced code used `?: "Unknown error"`. Restore a null fallback.
- Non-transactional failure persistence creates partial-write risk (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/workflow/PhotoEditExecutionPipeline.kt): `persistFailure` in `PhotoEditExecutionPipeline.kt` issues three separate repository calls (`savePhotoEditRequest`, `savePromptHistoryEntry`, `updateUpdatedAt`) with no surrounding transaction. The success path uses a single `queries.transaction {}`. A crash mid-failure-record can leave an incomplete failure state.
- `updateSelectedMediaAsset` accepts any existing media asset id, not just draft-associated ones (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/storage/SqlDelightManualWorkflowRepository.kt): The repository method only verifies the draft exists and then writes `selected_media_asset_id` directly, with no check that the asset belongs to the draft or its photo-edit results. The FK constraint enforces existence but not draft-association. Current UI callers pass valid ids, but the API permits invalid state.
- `selectEditedPhoto`/`selectOriginalPhoto` mutate terminal drafts (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/workflow/ManualPostDraftWorkspaceViewModel.kt): Both methods persist a selection and advance `updatedAt` unconditionally without checking draft status. Direct calls on `Archived` or `Shared` drafts silently mutate state. Other mutation paths in the ViewModel check status first; this is an inconsistency.
- No `savePhotoEditSuccess` coverage in `SqlDelightManualWorkflowRepositoryTest` (shared/src/androidUnitTest/kotlin/com/digitumdei/shotquill/shared/storage/SqlDelightManualWorkflowRepositoryTest.kt): The only transactional production path — `savePhotoEditSuccess` — is exercised only against `FakeManualWorkflowRepository` in pipeline tests. The real SqlDelight repository test has zero coverage of it, while the analogous `recordPostTextGeneration` is tested. SQL query or transaction bugs in `savePhotoEditSuccess` would not be caught.
- Redundant prompt: ViewModel sends raw wire-value instruction alongside structured assembler output (shared/src/commonMain/kotlin/com/digitumdei/shotquill/shared/workflow/ManualPostDraftWorkspaceViewModel.kt): The ViewModel passes `"Edit the image (${intent.wireValue}, ${platform.wireValue})."` as `userPrompt`, which the assembler appends on top of a structured prompt that already encodes `intent.promptIntent`, realism level, quality tier, and platform framing. The wire-value restatement adds no information and inflates token usage.
